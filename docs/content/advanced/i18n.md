title: Internationalization
---

Alloy supports multilingual sites through the `languages:` config key. Each language gets its own content directory, output path, and translation strings.

```yaml
# alloy.config.yaml
languages:
  en:
    title: "My Site"
    weight: 1
    root: true
    strings:
      read_more: "Read more"
      posted_on: "Posted on"
  fr:
    title: "Mon Site"
    weight: 2
    strings:
      read_more: "Lire la suite"
      posted_on: "Publie le"
```

Without the `languages:` key, Alloy builds a single-language site and none of the i18n features activate.

## Content structure

Each language has its own subdirectory under `content/`:

```
content/
├── en/
│   ├── index.md
│   ├── about.md
│   └── blog/
│       └── hello-world.md
└── fr/
    ├── index.md
    ├── about.md
    └── blog/
        └── hello-world.md
```

Every page exists independently per language. There is no fallback -- if `content/fr/about.md` does not exist, the French site has no about page.

## Output paths

By default, each language outputs under its language code prefix:

```
_site/
├── en/
│   ├── index.html
│   └── blog/hello-world/index.html
└── fr/
    ├── index.html
    └── blog/hello-world/index.html
```

### Root language

Set `root: true` on one language to output its pages at the site root instead of under a prefix:

```yaml
languages:
  en:
    root: true
    # ...
```

With this, English pages output to `_site/index.html` instead of `_site/en/index.html`. All other languages keep their prefix. Only one language should have `root: true`.

## Translation strings

Define translation strings in each language's config block and access them in templates through `site.language.strings`:

```liquid
<a href="{{ post.url }}">
  {{ site.language.strings.read_more }}
</a>

<time datetime="{{ post.date | date: '%Y-%m-%d' }}">
  {{ site.language.strings.posted_on }} {{ post.date | date: '%B %d, %Y' }}
</time>
```

The current language code is available as `{{ site.language.code }}`, which is useful for the `<html lang>` attribute:

```liquid
<html lang="{{ site.language.code }}">
```

## Shared layouts

All languages share the same `layouts/` directory. You do not create per-language layouts. Use `site.language.strings` and `site.language.code` to handle language-specific text within a shared layout:

```liquid
<!DOCTYPE html>
<html lang="{{ site.language.code }}">
<head>
  <title>{{ page.title }} — {{ site.language.title }}</title>
</head>
<body>
  <nav>
    {% for translation in page.translations %}
      <a href="{{ translation.url }}"
         {% if translation.lang == site.language.code %}aria-current="page"{% endif %}>
        {{ translation.lang }}
      </a>
    {% endfor %}
  </nav>
  <main>{{ content }}</main>
</body>
</html>
```

## Language switching

Every page has a `page.translations` array containing the alternate-language versions of that page. Each entry has `lang`, `url`, and `title` fields:

```liquid
<ul class="language-switcher">
  {% for translation in page.translations %}
    <li>
      <a href="{{ translation.url }}">{{ translation.lang }}</a>
    </li>
  {% endfor %}
</ul>
```

Pages are matched across languages by their path relative to the language directory. `content/en/about.md` and `content/fr/about.md` are translations of each other.

## Per-language collections and taxonomies

Collections and taxonomies are scoped per-language automatically. A collection defined globally applies to each language's content independently:

```yaml
collections:
  blog:
    permalink: "/blog/:slug/"
    sort: "date"
    reverse: true
```

This creates separate blog collections for each language. Iterating `collections.blog` in a template only yields posts from the current language. Taxonomy pages (tags, categories) are also generated per-language, so `/en/tags/go/` and `/fr/tags/go/` are independent pages listing only posts in their respective language.
